为什么git commit要规范？
我相信大部分人深有体会，公司有规范的您可以飘过，没有的话，特别是新入职的童鞋在阅读前辈们的git log时，说多了都是辛酸泪啊。
所以，很有必要git commit规范化！
核心目的是统一团队的Git Commit日志标准， 便于后续代码review和快速查找问题。

参考日志规范来源：angular

Commit Message日志基本规范

1
2
3
4
5

<type>(<scope>): <subject>
<BLANK LINE> //空一行
<body>
<BLANK LINE> //空一行
<footer>

type是必选项，scope可选项，subject是必选项，body和footer可选项。

Type

代表每次提交的类型，具体有如下：

类型 描述
feat 新增功能
fix 修复bug
improvement 对当前功能的改进
docs 仅仅修改了文档
style 不影响代码逻辑的修改(空格、格式等)
refactor 代码重构，既没新增功能也没修复bug
perf 优化相关，比如提升性能、访问速度、体验等等
test 测试用例，比如单元测试、集成测试等
chore 构建过程或辅助工具的变动
revert 回滚到上一个版本
build 影响构建系统或外部依赖项的更改(例如作用域:gulp、broccoli、npm)
ci 对CI配置文件和脚本的更改(示例范围:Travis、Circle、BrowserStack、SauceLabs)

Scope

用来说明本次Commit的影响范围，即简要说明修改会涉及的部分，比如应用层、模型层等，可为空。

Subject

本次Commit的简短描述

Body

对本次Commit的详细描述，可分为多行，可为空

Footer

分两部分

  • 关闭issue【针对开源项目】

如果当前commit是针对某个issue，此次修改关闭当前commit，例如 Close #12345

  • 不兼容变动

如果当前代码与上一个版本不兼容，则 Footer 部分以BREAKING CHANGE开头，后面是对变动的描述、以及变动理由和迁移方法

开源工具

你不会想着手动git commit吧，如果是的，恭喜啊，造轮子舒服吗？哈哈哈哈。去同性交友网站找个开源工具吧，比如Commitizen

安装使用Commitizen

1
2
3

  npm install -g commitizen
  npm init --yes
  commitizen init cz-conventional-changelog --save --save-exact

安装完毕，使用git cz代替git commit命令即可

全局安装

在根目录创建.czrc文件，安装插件且写入cz适配器配置

npm install -g commitizen cz-conventional-changelog

echo ‘{ “path”: “cz-conventional-changelog” }’ > .czrc
